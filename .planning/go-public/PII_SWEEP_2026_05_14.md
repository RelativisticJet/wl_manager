# Pre-Public PII / Real-Environment-Reference Sweep

> **Status**: Complete (2026-05-14).
> **Verdict**: PASS — no real PII, no real-environment references, no
> embarrassing artifacts in repo or git history.
> Two minor non-blockers noted in §3.

---

## 1. Why this sweep exists

Phase 0.10 of `docs/PUBLIC_RELEASE_PLAN.md` runs `git log --all -p`
secret-scan over commits — that catches API keys, passwords, tokens
in history, but it does NOT scan **test-data fixtures and screenshots
for real-environment references** (real customer hostnames, real
employee emails, real IPs, real PII). The Phase 0.0 AppInspect run
caught one such leak by accident — `lookups/_trash/DR999_stress_test.csv`
shipped `10.0.0.1`–`10.0.0.8` in the `.spl` via a package.sh
exclusion gap (now closed in build 660).

This sweep covers what Phase 0.10 misses: the fixture surface area
that lives in git but never gets scanned for non-secret-but-still-
sensitive content.

## 2. Scope and method

Scanned: every text-mode file under `lookups/`, `tests/`, `default/`,
`bin/`, `appserver/`, `docs/`. Excluded: `node_modules/`,
`.planning/appinspect/` raw reports.

Patterns searched:
- RFC 1918 private IPs (`10.x`, `172.16-31.x`, `192.168.x`)
- Email addresses
- Real-looking vendor domains (Google, Microsoft, CrowdStrike, etc.)
- "Firstname.Lastname" name patterns (real-people-adjacent)
- Tracked PNG screenshots

## 3. Findings

### 3.1 Private IPs in `lookups/DR*.csv` test fixtures — SYNTHETIC, accepted

Files like `lookups/DR45_whitelist_hosts.csv`,
`lookups/DR998_column_stress.csv`, etc. contain hundreds of `10.x.x.x`
addresses. Every host follows the synthetic pattern:

```
PROXY-428.dmz.local, 10.66.41.78, ad:0b:25:f7:05:3a, Ubuntu 22.04, HQ-Floor5, ...
POS-158.prod.internal, 10.218.219.20, d0:49:1d:10:58:a4, Ubuntu 22.04, Branch-Tokyo, ...
```

Characteristics that confirm synthetic origin:
- Hostnames use generic role-prefixes (`PROXY-`, `POS-`, `DNS-`,
  `FW-`, `SCAN-`, `VPN-`) plus a 3-digit number
- DNS suffixes are placeholder (`.dmz.local`, `.prod.internal`,
  `.hq.local`, `.lab.internal`, `.corp.local`)
- MAC addresses are randomly generated (no real-vendor OUI patterns
  beyond what random chance would produce)
- Office locations are generic (`HQ-Floor5`, `Branch-Tokyo`,
  `Branch-London`, `Branch-Sydney`)
- Owner teams are generic (`Cloud-Ops`, `IT-Infra`, `SOC`,
  `Network-Team`)

**Status**: All synthetic. None map to any real customer, employee,
or operational environment. Safe to publish.

**Also**: these files are EXCLUDED from `.spl` via
`scripts/package.sh --exclude='wl_manager/lookups/DR*'`, so they only
exist in git history, not in the installed app. The public
repo-history exposure is the only concern, and synthetic data is
explicitly safe for that.

### 3.2 Email in `docs/PUBLIC_RELEASE_PLAN.md` — INTENTIONAL, accepted

The decision document carries `communicate.oleh@gmail.com` (D15
record) and the `RelativisticJet` GitHub handle. This is explicitly
the user's chosen public-facing identity for Splunkbase publishing
(see D5 + D15 decision text in the same file). It is NOT inadvertent.

**Status**: Accepted per D15 explicit decision.

### 3.3 Vendor domains in `tests/a11y/reports/*.json` — Splunk own UI

A11y audit reports capture rendered HTML, including Splunk's own
"Splunk Answers", "Contact Support", "File a Bug", "Privacy Policy"
menu items linking to `community.splunk.com` and `www.splunk.com`.

**Status**: Splunk's own dashboard chrome. Not PII. Accepted.

### 3.4 Synthetic email patterns in `DR998_column_stress.csv` — SYNTHETIC

The stress-test CSV has columns like `email_sender` and
`email_recipient`. Sample values:

```
svc-eljl@corp.local, ftvb@external.com, fpks@corp.local, ...
```

Prefixes are 4-char gibberish, suffixes are placeholder domains. All
synthetic. Same exclusion as §3.1 — not shipped in `.spl`.

**Status**: Synthetic. Accepted.

### 3.6 No "Firstname.Lastname" matches

A scan for `\b[A-Z][a-z]{2,}\.[A-Z][a-z]{2,}\b` (e.g.
`John.Smith`, `Sarah.Connor`) in fixtures returned zero hits. Test
data uses pseudonymous patterns like `jsmith_EDIT`, `dwong_li_EDIT`,
`ijohnson_jr_EDIT` (suffixed with `_EDIT` and snake_case, not
firstname.lastname). All clearly synthetic.
