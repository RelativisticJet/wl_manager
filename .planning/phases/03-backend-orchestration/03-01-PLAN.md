Create bin/wl_filelock.py as a shared file locking utility module. This module provides a single file_lock(path, timeout) context manager used by all Phase 3+ modules (approval queue, limits config, versions, etc.).

**Module structure:**
- Import: contextlib, fcntl (with try/except for Windows), os, sys, threading, time
- Define RLock at module level: _file_lock_thread_lock = threading.RLock()
- Define context manager file_lock(path, timeout=10) yielding exclusive access

**Implementation details (from RESEARCH.md proven pattern):**
1. Use file_lock as a context manager that combines in-process RLock + cross-process fcntl.flock
2. Take path (file to lock) and timeout (seconds) as parameters
3. Acquire RLock first (prevent multiple threads in same process from competing for same lock)
4. If fcntl available (Unix), open lock file in append mode, acquire LOCK_EX, release on exit
5. If fcntl not available (Windows), no-op (Windows is dev-only; production is Linux)
6. On timeout or error, raise exception and fail-closed
7. Return tuple (can_lock: bool, remaining_timeout: float) for testing
8. Ensure lock file is cleaned up properly in finally block

**Function signature:**
```python
@contextmanager
def file_lock(lock_path: str, timeout: float = 10) -> Iterator[bool]:
    """Acquire exclusive lock on file with timeout. Yields True if lock acquired, False if Windows."""
    # Implementation...
```

**Key behaviors:**
- No-op on Windows (dev-only platform)
- 10-second timeout with 0.1-second retry interval (100 attempts)
- Proper finally cleanup even on exception
- Works with any file path (approval_queue.json, daily_limits.json, etc.)
- Thread-safe: in-process RLock prevents same-process contention

**Public exports:** __all__ = ["file_lock"]

**Type hints:** All parameters and return values annotated. No untyped code.

**Zero side effects:** No module-level code runs except constant definitions and logger setup.

Create bin/wl_limits.py to manage daily usage limits, reset scheduling, and enforcement. This module centralizes all limit checking and config management, replacing inline limit code in wl_handler.py (Phase 4 refactoring).

**Module responsibilities:**
1. Track daily usage per analyst per action-type (save_csv, delete_rule, etc.)
2. Enforce maximum action counts per day/week/month
3. Reset counters at configured UTC boundary times
4. Provide exemptions for admin users
5. Manage limit configuration (max counts for each action type)
6. Provide status API for frontend progress display

**File locations (in lookups/ directory):**
- daily_limits.json: usage counters {period_key: {user: {action_type: count}}}
- limit_config.json: configuration {action_type: max_count, approval_threshold: count, reset_time_utc: "HH:MM", reset_frequency: "daily/weekly/monthly"}

**Public functions (7 total, each ≤100 lines):**

1. check_analyst_limit(user: str, action_type: str, action_count: int = 1, roles: list = None) -> tuple(bool, int, int):
   - Check if analyst can perform action_count of action_type
   - Return (allowed: bool, current: int, max: int)
   - If roles includes admin: return (True, 0, -1) — admins exempt
   - If max_count == 0: return (False, current, 0) — disabled
   - If max_count == -1: return (True, current, -1) — unlimited
   - Otherwise: return ((current + action_count) <= max_count, current, max_count)
   - Use _get_counter_period_key() to bucket counters by reset frequency

2. check_admin_limit(user: str, action_type: str, action_count: int = 1) -> tuple(bool, int, int):
   - Separate function for admin-specific limits (may have different thresholds)
   - Same signature and return as check_analyst_limit
   - Use admin limit config instead of analyst config

3. get_limit_status(user: str, roles: list = None) -> dict:
   - Return {action_type: {current: int, max: int, remaining: int}} for all action types in config
   - Enables frontend to display usage progress and thresholds
   - For exempt users (admin), return -1 for max/remaining (signal as unlimited)

4. increment_daily_limit(user: str, action_type: str, amount: int = 1) -> bool:
   - Increment usage counter for user/action_type by amount
   - Called after action is successfully performed (in handler or approval flow)
   - Lock file during read→modify→write cycle
   - Return True on success, False on error (non-blocking)

5. set_limit_config(config: dict) -> tuple(bool, str):
   - Write new limit configuration atomically
   - Validate all required keys present and values are non-negative ints
   - Use temp+rename pattern with file_lock context manager
   - Return (success: bool, error_msg: str) tuple
   - Config shape: {action_type: int, approval_threshold: int, reset_time_utc: "HH:MM", reset_frequency: "daily"}

6. reset_daily_limits(analyst: str = None) -> tuple(bool, dict):
   - Reset counters for specified analyst (or all if analyst=RESET_ALL_USERS)
   - Called by scheduled search in handler on reset_time_utc boundary
   - Return (success: bool, reset_summary: dict) where summary = {analyst: reset_count}
   - Lock file during modification

7. get_limit_error_msg(user: str, action_type: str, current: int, max_int: int) -> str:
   - Format human-readable error message for limit exceeded
   - Example: "You have 3 edits remaining for today. Your request requires 5 edits."
   - Called by handler to include in error response

**Helper functions (private, support above):**

- _read_daily_limits() -> dict: Read daily_limits.json, fail-closed on corruption
- _write_daily_limits(counters: dict) -> bool: Atomic write with lock
- _read_limit_config() -> dict: Read limit_config.json with defaults fallback
- _get_counter_period_key() -> str: Return period bucket (e.g., "2026-04-01" for daily, "2026-W13" for weekly)
- _should_reset_now(reset_time_utc: str, reset_frequency: str) -> bool: Check if boundary crossed
- _get_limits_dir() -> str: Return lookups/ directory, create if missing

**Atomic write pattern (from Phase 2):**
