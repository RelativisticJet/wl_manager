**Public API:**

1. `get_counter_period_key(tz_offset_minutes: int = 0) -> str` — Return the counter period key for today (ISO format YYYYMMDD or "permanent"). Use tz_offset_minutes to adjust date boundary (default 0 = UTC).

   Examples:
   - tz_offset_minutes=0, current_utc=2026-03-31 20:30:00 → "20260331"
   - tz_offset_minutes=-300 (EST), same UTC time → "20260331" (date doesn't change)
   - tz_offset_minutes=-600 (CST), current_utc=2026-03-31 04:00:00 → "20260330" (date changed back by 1)

   Return string key suitable for use as dict key in counters.

2. `read_daily_limits() -> Dict` — Read daily limits counters from DAILY_LIMITS_FILE. Return (counters_dict). On file missing or JSON error, log warning and return empty dict.

   File format:
   ```json
   {
     "20260331": {
       "analyst1": {"row_removal": 25, "row_addition": 10, ...},
       "analyst2": {"row_removal": 5, ...}
     }
   }
   ```

3. `write_daily_limits(counters: Dict) -> Tuple[bool, str]` — Write counters to DAILY_LIMITS_FILE atomically. Return (success, error_msg).

4. `get_limit_config() -> Dict` — Read limit configuration from LIMITS_CONFIG_FILE. Return dict with keys: role_limits (dict), custom_limits (dict). On file missing, return default config.

   File format:
   ```json
   {
     "role_limits": {
       "analyst": {"row_removal": 50, "row_addition": 50, ...},
       "power": {"row_removal": 100, ...},
       "admin": {"row_removal": 999, ...}
     },
     "custom_limits": {}
   }
   ```

5. `set_limit_config(config: Dict) -> Tuple[bool, str]` — Write limit config to LIMITS_CONFIG_FILE atomically. Return (success, error_msg).

6. `check_daily_limit(user: str, action_type: str, action_count: int = 1) -> Tuple[bool, int, int]` — Check if user has room for action_count more items under the daily limit. Return (allowed_bool, current_count, max_count).

   Logic:
   - Read counters and limit config
   - Get today's period key
   - Look up user's current count for action_type
   - Look up max count from config (fallback to DEFAULT_LIMITS)
   - Return (current + action_count <= max, current, max)

7. `check_admin_daily_limit(admin_user: str, action_type: str, action_count: int = 1) -> Tuple[bool, int, int]` — Check if admin has room for approvals. Return (allowed_bool, current_count, max_count).

   Logic identical to check_daily_limit but uses "admin_approval_count" action_type and looks up max in admin tier limits.

8. `increment_daily_limit(user: str, action_type: str, count: int = 1)` — Increment the daily limit counter for a user and action type. Thread-safe via RLock.

   Steps:
   1. Acquire _limits_lock
   2. Read counters
   3. Clean up old period keys (keep only current and future)
   4. Add user bucket if missing (check MAX_TRACKED_ANALYSTS overflow)
   5. Increment counter[today][user][action_type] += count
   6. Write counters atomically
   7. Release lock

9. `increment_admin_daily_limit(admin_user: str, action_type: str, count: int = 1)` — Increment admin counter. Same logic as increment_daily_limit but tracks admin counters separately.

10. `reset_daily_usage(analyst: str, admin_user: str) -> Tuple[bool, str]` — Admin-initiated reset of analyst's daily counters. Return (success, error_msg).

    Steps:
    1. Acquire _limits_lock
    2. Read counters
    3. Find analyst in current period
    4. Delete analyst's counters for today
    5. Write counters
    6. Log reset with admin_user and timestamp
    7. Return (True, "")

11. `get_daily_limit_status(user: str) -> Dict` — Return current limit status for a user. Return dict with keys: role, limits, current_usage, remaining.

    Structure:
    ```python
    {
      "period": "20260331",
      "limits": {
        "row_removal": 50,
        "row_addition": 50,
        ...
      },
      "current_usage": {
        "row_removal": 25,
        "row_addition": 5,
        ...
      },
      "remaining": {
        "row_removal": 25,
        "row_addition": 45,
        ...
      }
    }
    ```

**Error Handling:**
- read_daily_limits: return empty dict on file missing or JSON invalid (non-breaking)
- write_daily_limits: return (False, error_msg) on permission denied or write failure
- get_limit_config: return default config on file missing
- set_limit_config: return (False, error_msg) on write failure
- All public functions: no exceptions; all errors returned in tuple or logged

**Thread Safety:**
- All file operations protected by _limits_lock (RLock allows reentrant acquisition)
- Check-then-increment is atomic via lock
