Create bin/wl_approval.py to manage approval queue operations, conflict resolution, and request submission/expiration. This module orchestrates Phase 1 (RBAC, validation) and Phase 2 (audit) modules.

**Module responsibilities:**
1. Read/write approval_queue.json with validation
2. Query pending requests (by CSV, by rule)
3. Submit new requests (single action, dual-admin) with validation and notification
4. Check and cancel conflicting requests
5. Expire pending requests older than APPROVAL_EXPIRY_DAYS
6. Auto-cancel requests when their prerequisites are deleted/modified

**File location:** lookups/approval_queue.json
Schema: Array of entries, each with:
```json
{
  "request_id": "unique-id",
  "status": "pending|approved|rejected|expired|cancelled",
  "timestamp": 1234567890,
  "analyst": "username",
  "action_type": "save_csv|delete_rule|create_rule|...",
  "csv_file": "DR123_whitelist.csv",
  "rule_name": "rule_name (if applicable)",
  "payload": {...action-specific data...},
  "reason": "approval reason",
  "resolved_by": "admin_name (if resolved)",
  "resolved_at": 1234567890,
  "cancelled_by_action": "action_type (if auto-cancelled)",
  "cancelled_by_analyst": "analyst_name (if auto-cancelled)"
}
```

**Public functions (8 total, each ≤100 lines):**

1. get_pending_for_csv(csv_file: str) -> list:
   - Return all non-expired pending requests for specific CSV file
   - Called during CSV operations to check if edits are gated
   - Filter by csv_file == target AND status == "pending" AND not expired
   - Call expire_pending_approvals() first to clean up old entries

2. get_pending_for_rule(rule_name: str) -> list:
   - Return all non-expired pending requests for specific rule
   - Same as above but filter by rule_name instead of csv_file

3. submit_approval(user: str, action_type: str, payload: dict, reason: str, roles: list, notify_fn=None) -> tuple(bool, str, dict):
   - Validate submission: user, action_type, payload required fields, reason length
   - Check approval gate: check_approval_gate(user, action_type, action_count, roles) — returns (needs_approval, limit_error)
   - If needs_approval: write to queue, trigger notification, return (True, "", entry)
   - If not needs_approval: return (False, "", entry) — direct approval (no queue entry, bypass)
   - Lock queue during write
   - Call notify_admins if approval needed and notify_fn provided
   - Return (success: bool, error_msg: str, entry: dict)
   - Validation errors fail-closed: return (False, error_msg, {})

4. submit_dual_approval(analyst_user: str, approver_user: str, action_type: str, payload: dict, reason: str, roles: list, notify_fn=None) -> tuple(bool, str, dict):
   - Special case: two-admin approval gate
   - Create entry with dual-admin metadata
   - Return same as submit_approval

5. check_approval_gate(user: str, action_type: str, action_count: int, roles: list) -> tuple(bool, str):
   - Check if action needs approval (calls wl_limits.check_approval_gate)
   - Return (needs_approval: bool, limit_error_msg: str)
   - Called by submit_approval to gate submissions

6. expire_pending_approvals(queue: list = None) -> list:
   - Remove entries older than APPROVAL_EXPIRY_DAYS from queue
   - Also prune resolved entries older than 30 days
   - Called at start of get_pending_for_* to clean up stale entries
   - Return modified queue (caller responsible for write)
   - Use int(time.time()) for timestamps, compare with <= not <

7. check_conflicts(queue: list, action: dict) -> list:
   - Dry run: return list of entries that WOULD be cancelled by this action
   - Does NOT modify queue or notify
   - Useful for testing and future UI preview
   - Matches actions like "delete CSV DR123.csv" against queue entries for that CSV
   - Logic: if action is delete_rule, cancel all pending edits for that rule; if delete_csv, cancel all pending edits for that CSV; etc.

8. cancel_conflicts(queue: list, action: dict, notify_fn=None) -> tuple(list, list):
   - Actually cancel conflicting entries
   - Return (modified_queue, cancelled_entries)
   - Caller responsible for writing modified queue
   - Call notify_analyst for each cancelled entry if notify_fn provided
   - Audit trail: each cancellation logs action='approval_cancelled', includes cancelled_by_action and cancelled_by_analyst

**Helper functions (private, support above):**

- _read_approval_queue() -> tuple(list, str): Read queue JSON with validation; return (entries, error_msg) on corruption
- _write_approval_queue(queue: list) -> tuple(bool, str): Atomic write with lock; return (success, error)
- _validate_queue_entry(entry: dict) -> tuple(bool, str): Check required fields present; return (valid, error_msg)
- _get_approval_queue_path() -> str: Return lookups/approval_queue.json path
- _generate_request_id() -> str: Generate unique request ID (uuid4)
- _is_expired(entry: dict) -> bool: Check if entry older than APPROVAL_EXPIRY_DAYS

**Atomic file operations (from Phase 2 pattern):**
```python
path = _get_approval_queue_path()
temp_path = str(path) + ".tmp"
try:
    with file_lock(path, timeout=10):
        with open(temp_path, "w", encoding="utf-8") as fh:
            json.dump(queue, fh, indent=2)
    os.replace(temp_path, path)
except OSError as e:
    return False, f"Failed to write queue: {e}"
```

**Conflict resolution logic:**
- If approved action is "delete_csv DR123.csv": cancel all pending edits/deletes for that CSV
- If approved action is "delete_rule Rule1": cancel all pending actions for that rule
- If approved action is "restore_csv": cancel pending "create CSV" requests for the same name
- Check on EVERY approval (not just destructive actions) to catch edge cases like approving edit after CSV was modified
- Functional style: return new queue copy, don't mutate input

**Precondition validation:**
- During approval replay (in handler), validate CSV/rule still exists
- If not: audit log with action='approval_precondition_failed', skip replay, continue to next queued action
- Prevents stale requests from silently failing or corrupting state
