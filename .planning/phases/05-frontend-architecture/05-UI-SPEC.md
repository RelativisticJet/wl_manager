## Frontend Architecture

### Module Structure

Phase 5 extracts the monolithic `whitelist_manager.js` (6,786 lines) into ~10-11 AMD modules organized by layer:

#### Foundation Layer (no inter-dependencies, no feature modules allowed as dependents)

| Module | Lines | Responsibility | Public API |
|--------|-------|---|---|
| `wl_constants.js` | ~100 | DOM selectors, config values, regex patterns, magic numbers | `{SELECTORS, CONFIG, PATTERNS, ...}` |
| `wl_state.js` | ~250 | Centralized application state manager with event-driven mutations | `{get, set, reset, batch, isDirty, on, off, register}` |
| `wl_rest.js` | ~150 | Shared REST helpers for HTTP calls to backend | `{restGet, restPost, setErrorHandler}` |
| `wl_ui.js` | ~100 | UI feedback utilities and theme management | `{showMsg, showFatalError, toggleTheme, init}` |

#### Feature Layer (depend on foundation, never on each other; communicate via State + custom events)

## State Manager Contract

### wl_state.js API

**Initialization:**
```javascript
State.register(key, defaultValue, validator);
// Validator: function(value) => void or throws TypeError on invalid
// Called once per app load before any features init
```

**Mutations:**
```javascript
State.get(key)              // Returns current value; throws TypeError on unknown key
State.set(key, value)       // Calls validator, updates, fires 'state:keyName' event; throws TypeError on validation fail
State.batch(updates)        // {key: value, ...}; applies all at once, fires events after all set
State.reset()               // Clears all shared state to registered defaults; fires 'state:reset' event
State.isDirty()             // Computed property: currentRows !== originalRows; auto-fires 'state:dirty' when changes
```

**Subscriptions:**
```javascript
State.on(event, callback)   // Events: 'state:keyName' or 'state:reset' or 'state:dirty'
State.off(event, callback)  // Unsubscribe
```

**Shared State Keys (locked set):**
- `currentRows` (Array<Object>) — Working rows in table, mutated by edit/add/remove operations
- `originalRows` (Array<Object>) — Rows as loaded from backend; immutable copy for dirty tracking
- `selectedRows` (Set<rowId>) — Row IDs selected via checkboxes
- `detectionRuleSelected` (String) — Name of currently selected detection rule (from dropdown)
- `csvFileSelected` (String) — Name of currently selected CSV file
- `pageIndex` (Number) — Current page number in paginated table (module-local state stays in wl_table.js)
- `columnWidths` (Object) — {columnName: width_px}; persisted to backend via save_col_widths
- `isDirty` (computed) — currentRows !== originalRows, fires 'state:dirty' event on change
- `pendingApprovalCount` (Number) — Count of pending approval requests for analyst
- `adminPendingCount` (Number) — Count of pending approvals awaiting admin review
- `userPresence` (Object) — {csv_file: {username: {role, idle_minutes}}}
- `notificationCount` (Number) — Unread notification count

**Module-Local State (never shared via State):**
- `wl_table.js`: `undoState`, `undoTimer`, `resizeState`, `dragState`, `currentPage`, `ROWS_PER_PAGE`, `searchQuery`
- `wl_modals.js`: `modalStack`, `activeModal`
- `wl_approval_ui.js`: `currentApprovalGate` (reason, action type)
- `wl_search.js`: `searchQuery`, `searchTimer`

## REST API Contract

### wl_rest.js Helpers

**HTTP Requests:**
```javascript
wl_rest.restGet(action, params)
  // action: string, e.g. 'get_csv', 'get_versions'
  // params: {key: value, ...} — converted to query string
  // Returns: jQuery.ajax promise with .done(callback) and .fail(callback)
  // Success: callback(response) where response = {csv: [...], ...}
  // Error: default handler fires 'wl:restError' event; .fail() callback receives XMLHttpRequest

wl_rest.restPost(action, payload)
  // action: string, e.g. 'save_csv', 'add_rule'
  // payload: object to send as JSON
  // Returns: jQuery.ajax promise
  // Success: callback(response)
  // Error: default handler fires 'wl:restError' with {status, responseJSON.error}

wl_rest.setErrorHandler(callback)
  // Register global error handler: callback({status, message, action})
  // Called by default .fail() unless specific .fail() registered
```

**Backend Actions (frozen contract, no changes):**
- GET: `get_csv`, `get_versions`, `get_csv_mapping`, `get_csv_list`, `get_rule_list`, `get_pending_approvals`, `get_audit_events`, `get_presence_status`
- POST: `save_csv`, `save_col_widths`, `revert_csv`, `add_rule`, `delete_rule`, `create_csv`, `delete_csv`, `process_approval`, `cancel_request`, `log_event`, `mark_notifications_read`, `restore_from_trash`, `purge_trash`, `set_trash_retention`, `reset_factory_defaults`, `save_as_default`

## Module Communication Protocol

### Custom Events (jQuery delegated, document-wide)

**Naming Convention:** `wl:actionName` for inter-module communication; `state:keyName` for state mutations

**Event Flow Patterns:**

1. **Feature → Orchestrator (Intent Delegation)**
   - `wl_table` fires `'wl:removeRequested'` with `{rowId}` → `whitelist_manager.js` calls `WlModals.showRemoveDialog(rowId)`
   - `wl_table` fires `'wl:ruleDeleteRequested'` with `{ruleName}` → orchestrator handles rule deletion flow
   - `wl_modals` fires `'wl:removalConfirmed'` with `{rowIds, reason}` → orchestrator calls `wl_rest.restPost('remove_rows', ...)`

2. **State Events (Reactive Updates)**
   - Any module sets `State.set('currentRows', newRows)` → fires `'state:currentRows'` → subscribed listeners re-render
   - Table sets `State.set('selectedRows', rowSet)` → fires `'state:selectedRows'` → approval button updates enabled state
   - Entry point listens to `'state:dirty'` → saves button visually changes

3. **Error Propagation**
   - `wl_rest.restPost()` fails → fires default `'wl:restError'` event → `wl_ui.showMsg('error', message)` displays toast
   - Module can override per-call with `.fail(customHandler)` for special cases (e.g., 409 conflict handling with merge prompt)

4. **Notifications & External State Changes**
   - `notifications.js` detects new notifications → fires `'wl:notificationsUpdated'` → orchestrator increments `State.set('notificationCount', ...)`
   - External CSV change detection → fires `'wl:externalChangeDetected'` → orchestrator prompts reload

## AMD Module Loading & Initialization Order

**RequireJS path resolution:** Relative paths within `appserver/static/modules/` (e.g., `require('modules/wl_rest')`)

**Initialization sequence (strict order):**

1. `wl_constants.init()` — register all constant values
2. `wl_state.init()` — register all shared state keys with defaults and validators
3. `wl_rest.init()` — set default error handler to `wl_ui.showMsg()`
4. `wl_ui.init()` — theme detection, message container setup
5. `wl_search.init()`, `wl_presence.init()`, `wl_csv_io.init()` — independent features (parallel safe)
6. `wl_table.init()`, `wl_modals.init()`, `wl_versions.init()`, `wl_approval_ui.init()` — coupled features (depend on table state)
7. `notifications.init()` — long-running notification polling
8. Orchestrator (`whitelist_manager.js`) — load initial CSV from URL params, wire cross-module event listeners

**Failure handling:** If any `init()` throws, orchestrator catches and calls `WlUI.showFatalError(message)`. No partial initialization.

## Testing Contract

### QUnit Framework (Frontend Testing)

**Test files location:** `tests/qunit/test_*.js`

**Test subjects (Phase 5 scope):**
1. `test_state_manager.js` — State registration, get/set validation, event firing, batch mutations, dirty detection
2. `test_rest_helpers.js` — REST request construction, error handling, default fail handler

4. `test_state_events.js` — State event subscriptions, cross-module event communication, event cleanup

**Assertions per test file:** ~12-15 per file, ~50 total across Phase 5

- Hidden Splunk dashboard `test_runner.xml` loads QUnit library and test files

- Vendored in `tests/qunit/` directory (not via npm)
