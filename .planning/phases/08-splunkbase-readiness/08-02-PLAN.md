Structure using STRIDE:
1. **Spoofing Identity**: How analyst identity is verified (Splunk user context, session validation)

6. **Elevation of Privilege**: How admin-only actions are gated (wl_approval.py, approval queue required)

Include DREAD scoring for top threats:
- Damage potential (1-10): How much harm if exploited
- Reproducibility (1-10): How easy to reproduce
- Exploitability (1-10): How much skill required
- Affected users (1-10): What percentage of users
- Discoverability (1-10): How easy to find the vulnerability

**Mitigated Threats Section**:
Include evidence of vulnerabilities identified and fixed during development:
1. Optimistic locking bypass: Fixed via expected_mtime validation in wl_versions.py
2. Client trust bypass: Fixed via server-side validation in wl_validation.py

**Data Flow Diagrams** (Mermaid format):
- Analyst action flow: User submits edit → validation → approval queue → audit event → CSV updated
- Authentication flow: Session key → Splunk user context → role lookup → permission check
- Audit event flow: Action → audit event construction → HTTP POST → wl_audit index

| admin | Yes | Yes | Yes | Yes | Yes | Yes | Yes | Yes |
| superadmin | Yes | Yes | Yes | Yes | Yes | Yes | Yes | Yes |

- Analyst identity from Splunk session key

- No cached permissions (checked at request time)
- Daily usage limits enforced (wl_limits.py)
- Approval gate for bulk edits and deletions (wl_approval.py)
- Complete audit trail in wl_audit index
