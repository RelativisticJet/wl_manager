PART 1: USER GUIDE

1. Introduction

Whitelist Manager is a Splunk application that provides a web-based interface for managing detection-rule CSV whitelists with a full audit trail. Instead of manually editing CSV lookup files on the filesystem, security analysts can use the dashboard to view, add, edit, and remove whitelist entries — with every change automatically tracked.

Key Capabilities

  • Searchable dropdown for 300+ detection rules
  • Editable CSV table with pagination (10 rows per page)
  • Full audit trail — who, what, when, and why for every change
  • Expiration management — date picker with presets; expired rows auto-cleaned
  • Git-style diff — see exactly what changed after each save
  • Cell-level edit tracking — before/after values for every field
  • CSV import/export — bulk upload and download
  • Role-based access control — editors vs. viewers
  • Dark/light theme — adapts automatically

2. Getting Started

Accessing the App

  1. Log in to Splunk Web
  2. Navigate to Apps > Whitelist Manager
  3. The main dashboard loads automatically

Required Roles

ActionRequired Role
View whitelists and audit trailwl_viewer, wl_editor, admin, or sc_admin
Edit whitelists (add/remove/modify)wl_editor, admin, or sc_admin

First-Time Workflow

  1. Select a Detection Rule from the dropdown
  2. Select a CSV File from the second dropdown
  3. The CSV contents load in an editable table
  4. Make your changes (add, edit, or remove rows)
  5. Click Save Changes
  6. Provide a comment explaining the change (required)
  7. Review the diff summary

3. Main Dashboard — Managing Whitelists

3.1 Selecting a Detection Rule

The Detection Rule dropdown supports type-ahead search. Start typing the rule name (e.g., “DR20” or “malicious”) and matching rules appear instantly. Click a rule to select it.

Once selected, the CSV File dropdown populates with all CSV files mapped to that rule.

3.2 Viewing CSV Contents

The table displays:

  • Row numbers (#) — visual 1-based row numbers
  • Checkboxes — for selecting rows
  • Data columns — all visible columns (internal _ columns are hidden)
  • Actions — a “Remove” button per row

The table paginates at 10 rows per page with First/Prev/Next/Last navigation.

3.3 Adding Rows

  1. Click + Add Row
  2. A new empty row appears at the bottom (auto-navigates to last page)
  3. Fill in all required fields
  4. Click Save Changes when done

3.4 Editing Rows

All cells are directly editable. Click a cell and type. Changes are tracked at the cell level with before/after values in the audit trail.

3.5 Removing Rows

Click the Remove button on any row. A prompt asks for a removal reason (required). The row is removed and saved immediately. A 10-second undo bar appears.

3.6 Bulk Removal

  1. Check boxes next to rows you want to remove (works across pages)
  2. Use the header checkbox to select/deselect all rows across all pages
  3. Click Remove Selected (N)
  4. Enter a removal reason (applies to all selected rows)

3.7 Undo Functionality

After removing a single row, a 10-second undo bar appears. Click Undo to restore the row. After 10 seconds, the option expires.

Note: Undo is available for single-row removals only.

3.8 Setting Expiration Dates

If the CSV has an expiration column, clicking the cell opens a date/time picker with presets:

PresetEffect
7 DaysSets expiration 7 days from now
30 DaysSets expiration 30 days from now
6 MonthsSets expiration ~182 days from now
1 YearSets expiration 365 days from now
Clear (Permanent)Removes expiration (row never expires)

Date format: YYYY-MM-DD HH:MM

Supported column names (case-insensitive): Expires, expire, expiration, expiration_date, expiry, termination, termination_date

3.9 Import and Export CSV

Export: Click Export CSV to download the current CSV content (metadata columns excluded).

Import: Click Import CSV and select a .csv file. Only new rows not already in the CSV are added. Review and click Save Changes to persist.

3.10 Discarding Changes

Click Discard Changes to revert all unsaved modifications to the last saved state.

5. Understanding the Diff View

After saving, a Change Summary appears below the table:

  • Added rows — green section with each new row
  • Removed rows — red section with each deleted row
  • Unified diff — Git-style view with + (added), - (removed), @@ (markers)

7. FAQ

Q: I don’t see any detection rules in the dropdown

The rule_csv_map.csv lookup needs to be populated. Contact your Splunk administrator.

Q: I get “Permission denied” when saving

You need the wl_editor role. Contact your Splunk administrator.

Q: Rows disappeared when I loaded the CSV

Rows past their expiration date are automatically removed on load. A yellow banner shows how many were removed.

Q: My expiration dates aren’t being recognized

Ensure the column name matches one of the supported names and the date is in YYYY-MM-DD HH:MM or YYYY-MM-DD format.

Q: How do I make a row permanent?

Leave the expiration column empty or use Clear (Permanent) in the date picker.

Q: Where is audit data stored?

In the wl_audit index (sourcetype wl_audit). A backup log is at $SPLUNK_HOME/var/log/splunk/wl_manager_audit.log.

9. Installation

Option A: Splunk Web

  1. Download the .spl file
  2. Go to Apps > Manage Apps > Install app from file
  3. Upload the file and restart Splunk

Option B: CLI

$SPLUNK_HOME/bin/splunk install app wl_manager-1.0.0.spl
$SPLUNK_HOME/bin/splunk restart

Option C: Manual

tar -xzf wl_manager-1.0.0.spl -C $SPLUNK_HOME/etc/apps/
chown -R splunk:splunk $SPLUNK_HOME/etc/apps/wl_manager/
$SPLUNK_HOME/bin/splunk restart

Post-Installation Verification

curl -sk -u admin:password \
  https://localhost:8089/services/custom/wl_manager?action=get_mapping

10. Initial Configuration

10.1 Custom Roles

Two roles are created automatically:

RoleInheritsPurpose
wl_editoruserRead + write whitelists, search wl_audit
wl_vieweruserRead-only access

Assign via Settings > Access Controls > Users or:

$SPLUNK_HOME/bin/splunk edit user <username> -roles user,wl_editor -auth admin:password

10.2 wl_audit Index

Created automatically. Default: 1 GB cap, 3-year retention. Verify:

| rest /services/data/indexes/wl_audit | table title currentDBSizeMB maxTotalDataSizeMB

10.3 Rule-to-CSV Mapping

Edit lookups/rule_csv_map.csv via Settings > Lookups or directly on the filesystem:

rule_name,csv_file,app_context

DR600_unauthorized_software,dr600_unauthorized_software.csv,

11. Adding Detection Rules and CSVs

  1. Create the CSV file with a header row in the appropriate lookups/ directory
  2. Add an entry to rule_csv_map.csv
  3. Ensure the Splunk user has read/write permissions on the CSV file: 
    chown splunk:splunk $SPLUNK_HOME/etc/apps/<app>/lookups/<csv>
chmod 644 $SPLUNK_HOME/etc/apps/<app>/lookups/<csv>
  4. Verify in the dashboard

12. Expiration Settings

The app detects these column names (case-insensitive) as expiration dates:

Expires, expire, expiration, expiration_date, expiry, termination, termination_date

Supported formats: YYYY-MM-DD HH:MM, YYYY-MM-DD

How it works:

  • On load: Expired rows removed automatically (uses browser timezone)
  • Hourly: Scheduled cleanup removes expired rows (uses server timezone)

13. Scheduled Cleanup

Configured in inputs.conf:

[script://./bin/wl_expiration_cleanup.py]
disabled       = false
interval       = 3600
python.version = python3

To adjust frequency, create local/inputs.conf with a different interval. Monitor with:

index=_internal sourcetype=wl_expiration_cleanup

14. Audit Trail & Retention

Event Structure

FieldDescription
timestampUnix epoch
analystUsername (“system” for auto-cleanup)
actionadded, removed, removed_multiple, edited, auto_removed
detection_ruleRule name
csv_fileCSV filename
commentAnalyst-provided reason
valuePer-row field values
remove_reasonWhy rows were removed

Useful SPL Queries

# All changes in last 24h
index=wl_audit sourcetype=wl_audit earliest=-24h
| table timestamp analyst action detection_rule csv_file

# Changes by analyst
index=wl_audit sourcetype=wl_audit
| stats count by analyst action | sort -count

Retention

Default: 3 years (94,608,000 seconds). Adjust in local/indexes.conf. Backup log at $SPLUNK_HOME/var/log/splunk/wl_manager_audit.log (10 MB rotating, 5 backups).

15. CLI Wrapper Usage

# List CSV contents
python3 bin/wl_wrapper.py list --csv dr20_whitelist.csv --app SplunkEnterpriseSecuritySuite

# Add a row
python3 bin/wl_wrapper.py add --csv dr20_whitelist.csv --app SplunkEnterpriseSecuritySuite \
  --rule DR20 --values "host=SRV1,user=bob,Comment=Approved" --user john --comment "INC-123"

# Remove rows
python3 bin/wl_wrapper.py remove --csv dr20_whitelist.csv --app SplunkEnterpriseSecuritySuite \
  --rule DR20 --match "host=SRV1,user=bob" --user john --comment "No longer needed"

# Dry-run diff
python3 bin/wl_wrapper.py diff --csv dr20_whitelist.csv --app SplunkEnterpriseSecuritySuite \
  --values "host=SRV2,user=alice"
Note: The CLI wrapper writes audit events to the log file only (no Splunk session key). For full audit coverage, prefer the web interface.

17. Troubleshooting

App not visible

Check: ls $SPLUNK_HOME/etc/apps/wl_manager/ and ownership. Restart Splunk.

REST endpoint returns 404

Verify restmap.conf and web.conf. Check: index=_internal source=*splunkd.log* wl_manager ERROR

Permission denied on CSV save

Fix file ownership: chown splunk:splunk <csv_path>

Dashboard shows stale data

Bump build number in app.conf and restart, or use: curl -k -u admin:pass https://localhost:8089/services/apps/local/wl_manager/_bump

Docker: files owned by root

Fix with: docker exec -u 0 <container> chown -R splunk:splunk /opt/splunk/etc/apps/wl_manager/

18. Upgrading & Uninstallation

Upgrading

  1. Back up local/ directory and lookups/rule_csv_map.csv
  2. Install the new .spl (overwrites default/)
  3. Restore your rule_csv_map.csv
  4. Clear Python cache: find ... -name "__pycache__" -exec rm -rf {} +
  5. Restart Splunk

Uninstallation

$SPLUNK_HOME/bin/splunk remove app wl_manager
$SPLUNK_HOME/bin/splunk restart

# Optional: remove the audit index
$SPLUNK_HOME/bin/splunk remove index wl_audit

# Optional: remove the log file
rm $SPLUNK_HOME/var/log/splunk/wl_manager_audit.log*
